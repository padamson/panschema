# Changelog

All notable changes to this project will be documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.0.0/),
and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [Unreleased]

### Added
- **New `json-schema` output format: `generate --format json-schema` emits a JSON Schema from the LinkML schema.** A schema's classes become a JSON Schema (draft 2020-12) — one closed `object` per class under `$defs`, with each class's effective slots as typed properties (LinkML ranges mapped to JSON Schema `type`/`format`, multivalued slots wrapped in `array`), a `required` list from effective cardinality, and `additionalProperties: false`. This is the language-agnostic structured-output contract: an LLM (via a framework like `rig`, or Anthropic/OpenAI structured output) can be constrained to produce valid instances, and any JSON validator can check instance data against the same schema — the JSON an LLM returns *is* a LinkML instance. Available on `generate --format json-schema` and as a `json_schema` output in a manifest's `[generate.<schema>]` block. Every emitted document is checked with a real, independent JSON Schema validator (the `jsonschema` crate): it must compile as a valid schema, and representative instances must validate as expected (well-formed data passes; missing-required, wrong-typed, and extra-property data all fail). Current coverage is scalar properties; enum/class `$ref`s, `pattern`/bounds constraints, and inheritance flattening are still to come.
- **Worked-example individuals now render as a separate instance graph.** An ontology's OWL `NamedIndividual`s (its A-box) draw as their own force-directed graph beneath the Individuals section of the HTML docs — nodes are individuals (teal hexagons, typed and carrying their literal assertions), edges are their object-property assertions between individuals. It is a *distinct* viz from the schema graph, not an overlay: the schema (T-box) graph stays uncluttered, and the instance (A-box) graph reads on its own, right where the individual cards are documented. The graph writer gains `schema_to_instance_graph` (a separate `Individual`-node/`Assertion`-edge graph built from the individuals panschema already ingests); it is emitted only when the schema actually declares individuals. Verified by reading a fixture ontology through the OWL reader and by an end-to-end browser test that asserts the instance canvas paints the individual nodes.
- **Cross-package `imports:` — a schema can import a manifest dependency by name.** An `imports:` entry that names a `[schemas]` dependency (rather than a local file) now resolves to that dependency's schema in its own package and merges it, so a layering application composes one or more shared schemas plus its own local classes into a single generated output. Resolution precedence is local file → manifest dependency → built-in, so a real local import is never shadowed by a same-named dependency; local imports stay contained within their package's directory, while a manifest-declared dependency is a trusted jump into its own tree. The imported dependency's prefixes union into the importing schema, so a cross-namespace CURIE resolves. An `imports:` entry that names neither a local file nor a declared dependency fails with a diagnostic naming the entry and pointing at `[schemas]` + `panschema fetch`, instead of a silent drop. An application can import several shared schemas at once — including a diamond where two of them import a common base, which merges once — and a class in the app resolves references to any imported class. When two dependencies define the same element *differently* and neither is the importing schema, the command fails naming both source files instead of silently picking one by import order; a root-vs-import override stays a deterministic warning (the importing schema wins).
- **Manifest-driven `generate` fans out to every writer, not just HTML and Rust.** A consumer project's `panschema.toml` `[generate.<schema>]` block now accepts `postgres`, `shacl`, `ttl`, `jsonld`, `rdfxml`, `ntriples`, and `graph-json` output paths alongside `html` and `rust`, and `panschema generate` (with no `--input`) emits each configured format for every manifested schema in one run — so a relationally-backed application gets its Postgres DDL from the same manifest that produces its Rust types. A writer that errors fails the command naming the schema and format it was producing; an unconfigured format is simply not emitted.
- **New `shacl` output format: `generate --format shacl` emits a SHACL shapes graph from the LinkML schema.** A schema's declared value-constraints are now machine-checkable by any SHACL engine, not just visible in the docs — one `sh:NodeShape` per class (targeting the same class IRI the OWL output declares) with a `sh:property` shape per slot carrying its constraints: range → `sh:datatype` (scalar) or `sh:class` (class-valued), `required` and `minimum_cardinality` reconciled into a single `sh:minCount` (explicit cardinality wins, so the two never contradict), `maximum_cardinality` → `sh:maxCount`, `pattern` → `sh:pattern`, and `minimum_value`/`maximum_value` → `sh:minInclusive`/`sh:maxInclusive`. A class `rule` (a conditional requirement like "when `status` = `actual`, `region` is required") becomes a conditional `sh:or ( [ sh:not <precondition-shape> ] <postcondition-shape> )` — the SHACL-Core analogue of an if/then constraint — with the equality checks projected as `sh:hasValue`. The output is a standalone Turtle shapes file (separate from the OWL/TTL output), and every generated graph is loaded into a real, independent triple store (`oxigraph`) in the tests to confirm the shape triples are well-formed and queryable. Cross-instance `unique_keys` uniqueness (which SHACL Core can't express without SPARQL) is still to come.
- **Generated HTML documentation is now checked against a real HTML5-conformance parser.** A browser silently repairs malformed markup, so passing the existing browser-based end-to-end tests was never proof the generated HTML was actually spec-valid. Every generated page is now parsed with the same HTML5 engine Servo/Firefox use, and any parse error (mis-nested tags, unclosed elements, and the like) fails the test suite. No output change for users — this only strengthens what confirms the output is correct.
- **Generated RDF/OWL output is now checked against a real, independent triple store.** Every RDF-family writer (TTL, JSON-LD, RDF/XML, N-Triples) shares one serialization core, so nothing previously confirmed the resulting graph was sane beyond panschema's own reader parsing it back. Generated TTL is now loaded into an in-memory `oxigraph` store and queried with SPARQL — e.g. confirming every class actually carries an `rdf:type owl:Class` triple and a declared `subClassOf` resolves to a real triple — catching graph-level mistakes a shared internal blind spot wouldn't self-report. No output change for users — this only strengthens what confirms the output is correct.
- **Generated Postgres DDL is now confirmed to apply against a real database.** Passing syntax validation only proves the output is well-formed SQL, not that it actually works — a foreign key referencing a column that doesn't exist, for instance, is syntactically fine but fails at apply time. A representative multi-class fixture (an enum, both kinds of primary key, a foreign key between them) is now applied against a disposable, real Postgres database in CI, catching that class of error. No output change for users — this only strengthens what confirms the output is correct.
- **New `postgres` output format: `generate --format postgres` emits Postgres DDL from the LinkML schema.** A relationally-backed application can now get its database schema from the same LinkML source as its Rust structs, with no hand-written SQL. Current coverage: one `CREATE TABLE` per concrete class with its effective scalar attributes as typed columns (a column is `NOT NULL` when the slot's effective lower bound requires it — `required`, or an explicit `minimum_cardinality ≥ 1`); enum-range slots get a `CREATE TYPE ... AS ENUM` declared before any table references it; the effective `identifier` slot becomes the primary key, or a synthesized `id uuid PRIMARY KEY DEFAULT gen_random_uuid()` when there isn't one; a single-valued class-range slot becomes a foreign key column named after the target's actual primary key column, added via a deferred `ALTER TABLE ... ADD CONSTRAINT` after every table is declared (so declaration order and cycles between classes never matter); mixin attributes flatten into every class that mixes them in, matching how the Rust writer already flattens them; an abstract class gets no table. Every emitted identifier — table, column, enum type, and constraint names — is double-quoted, so a class named `Order` (a SQL reserved word) produces applicable DDL, and a name crafted to escape the identifier position cannot alter the statement. Class- and slot-level constraints are now enforced by the database, not just documented: `unique_keys` become table-level `UNIQUE` constraints (a key naming a slot the class doesn't have is dropped rather than emitting a broken reference), a slot `pattern` becomes an inline `CHECK (col ~ '...')`, `minimum_value`/`maximum_value` become an inline `CHECK` bounding the column, and a class `rule` (a conditional requirement like "when `status` = `actual`, `region` is required") becomes a conditional `CHECK (NOT (precondition) OR (postcondition))` — a rule with no single-column `CHECK` form (one-sided, or a `range`/cardinality condition) is skipped with a warning naming it rather than emitted broken. A class using `is_a`, a multivalued slot, `any_of`, or referencing another skipped class is itself skipped with a warning naming why, rather than emitting broken DDL — multivalued columns and `is_a` support are not modeled yet. Every rendered fixture is checked against `pg_query` — a binding to Postgres's own C parser — confirming it's genuinely valid Postgres SQL, not just a Rust string that matches this codebase's own expectations, and a representative fixture is additionally applied against a real, disposable Postgres database in CI.
- **Class-level `unique_keys` now render on the class card.** A class's uniqueness constraints (`unique_keys:` — named tuples of slots whose combined values must be unique) previously parsed and silently vanished. The class card now shows a "Unique keys" row listing each key's slot tuple with its optional description. Each referenced slot is checked against the class's effective slot set (inherited, mixed-in, and refined slots included); a key naming a slot the class doesn't have produces a warning at generate time, so a typo'd or stale constraint is caught rather than rendered broken. Enforcement against instance data remains the consuming application's responsibility. Beyond the docs, `unique_keys` also project to Postgres `UNIQUE` constraints (see the `postgres` format entry); for formats that don't project them (`rust`, `ttl`, ...), generating warns that they won't appear in that output, naming the format actually requested, so the gap isn't silent.
- **Class-level `rules` now render on the class card.** A class's conditional constraints (`rules:` — precondition/postcondition pairs over `slot_conditions`) previously parsed and silently vanished. The class card now shows a "Rules" section: each rule's title/description plus a human-readable "when … then …" sentence built from its conditions (`range`, `required`, cardinality, value bounds, `pattern`, `equals_string` / `equals_number` equality checks, `value_presence` presence checks, and `any_of` alternatives — both a single slot's alternative values (`verdict` is `approved` or `rejected`) and alternative whole condition sets) — e.g. "when `status` = `actual`, then `region` is required", or "when `verdict` = `approved` or `verdict` = `rejected`, then `approved_by` is present". (`value_presence` and both `any_of` forms were previously parsed away and left a rule looking like a bare description — a decided-vs-pending rule showed its consequence with no trigger; all now round-trip in the IR and render.) The graph JSON export now carries each class's rules (title, description, and the same rendered summary) in its node metadata, where previously the graph omitted rules entirely — so the schema-graph hover can surface them. Each **slot** card now also gains a "Rules" section listing every class rule that references that slot (on either the trigger or governed side), naming the class and the rendered summary — so a reader on a conditionally-required slot sees the rule that constrains it; because the graph hover / pinned card reuses the slot's HTML card, the same section shows when you hover or click a slot node. Beyond the docs, `rules` also project to Postgres conditional `CHECK` constraints and SHACL conditional shapes (see those format entries); for formats that don't project them (`rust`, `ttl`, ...), generating warns that they won't appear in that output, naming the format actually requested, so the gap isn't silent.
- **Warns on silently-dropped LinkML class constructs.** panschema parses but does not model many class-level LinkML keys (e.g. `rules`, `unique_keys`, boolean class expressions); previously a schema could declare one and have it vanish from every output with no signal. `generate` now warns on **any** unmodeled class key by default — naming the key and the class (e.g. "`rules` on class `Deployment` is parsed but not modeled; it will not render or emit"). The ignore-list of keys that don't warn starts empty and grows only as a specific key is identified as safe to silence; warning by default (rather than an allowlist of known constructs) is what catches drops that weren't anticipated, including a producer's typo of a real field. `generate --strict` turns these warnings into a non-zero exit (both `--input` and manifest modes), so CI can fail a build that would silently drop a constraint. Output is otherwise unchanged — the construct is still dropped, just no longer silently.
- **`mdbook-panschema install` — a maintained mdbook→schema toolbar link.** A new `mdbook-panschema` binary installs a toolbar button that links an mdbook book to its panschema-generated schema docs, the way `mdbook-admonish install` drops its assets: `mdbook-panschema install [dir]` (`dir` defaults to the current directory) writes `schema-link.js` / `schema-link.css` into the book and wires them into `book.toml`'s `additional-js` / `additional-css`, idempotently (re-run after an upgrade to refresh). The button's link target and label come from a new opt-in `[book_link]` section in `panschema-publish.toml` — `enabled` (default `false`), `schema_path` (default `"schema/current/"`), and `label` (default `"Schema reference"`) — validated on load so a typo'd key or wrong value type fails loudly instead of silently reverting to a default. With `[book_link]` absent or `enabled = false`, `install` is a no-op. Replaces the per-book custom JS/CSS every consumer previously hand-rolled.
- **Local `imports:` now resolve and merge at load time.** A LinkML schema that lists other local files in `imports:` previously had those entries parsed and then ignored, so the schema rendered only the root file's elements. Each import naming a local file (resolved relative to the importing file; `.yaml`/`.yml`/`.ttl`, extension optional) is now loaded and folded into the root before any writer runs — its `classes`, `slots`, `enums`, `types`, and `prefixes` union into one schema, with the importing file winning on a name collision. Docs, codegen, RDF, and graph output all see a single self-contained schema, so a vocabulary split across files renders as one. Import resolution runs through one shared load path used by `generate`, `serve`, and `publish` alike, so a split schema also previews correctly under the dev server's hot reload and in published output — previously only `generate` resolved imports. That shared path also surfaces the schema-level diagnostics (unmodeled constructs, `unique_keys` naming a slot the class lacks) on all three commands, not just `generate`.
- **Dangling references now produce one clear warning instead of six silent degradations.** A slot `range`, a class `is_a` parent or `mixin`, or a slot `inverse` that names nothing the loaded schema defines — no class, enum, `types:` entry, or LinkML built-in primitive — previously degraded differently and silently in each writer: the graph dropped the edge, the RDF/SHACL writers minted an IRI from the bare name, Postgres fell back to `text`. Loading a schema now reports each unresolved reference once, naming the referrer and the missing name, on `generate`, `serve`, and `publish` alike; `generate --strict` turns it into a non-zero exit. Valid primitive ranges are never flagged (the full LinkML built-in type set resolves), and a schema's own `types:` entries resolve too. Imports resolve transitively (imports of imports), and a file reached through two paths — a diamond — is loaded and merged exactly once. When two files define the same name, identical definitions are silently unified while differing ones surface a warning naming both the kept and dropped source files (the importing file still wins). A self-import or an import cycle — direct or transitive — is reported as a clear error rather than looping. Built-in and remote imports are recognized and skipped as no-ops: the standard `linkml:*` modules (e.g. `linkml:types`), and any CURIE or URL that expands to a remote `http(s)` URI, name well-known schemas rather than local files, so a schema can import LinkML's built-in types the canonical way without codegen failing.
- **Vendored dogfood-schema regression fixtures + weekly release monitor.** Every released (tagged) version of the real dogfood schemas is checked in as a frozen snapshot under `tests/fixtures/dogfood/<repo>/<tag>.yaml` and exercised by panschema's own tests: each vendored release is read, rendered to Rust, and validated with `syn::parse_file`, and the latest release per schema is additionally `cargo build`-compiled in a scratch crate (older tags stay at the parse check to keep CI bounded). The suite runs offline — a codegen regression that would break a real schema fails CI with no network. `scripts/vendor-dogfood-schemas.sh <repo> <tag|all>` fetches a release and writes its snapshot (run by hand, committed explicitly), and a weekly workflow opens a single tracking issue when a release hasn't been vendored yet.
- **Schema-encoded slot defaults via `ifabsent`.** A non-multivalued slot with an `ifabsent` default now generates a non-`Option` Rust field that deserializes to that default when absent — the field renders as the bare type with `#[serde(default = "…")]` and a generated default fn, so the default lives in the schema instead of as a convention each consumer re-encodes by hand. Supported forms: the enum form `"Enum(value)"` (or the bare `"value"` resolved against the slot's range enum) maps to the matching variant; the scalar forms `int(N)`, `float(F)`/`double(F)`, `string(S)` (escaped), and boolean (`true`/`false`) map to the field's `i64`/`f64`/`String`/`bool` literal. An `ifabsent` that doesn't parse or resolve falls back to the `Option<T>` field with a warning comment rather than a broken default. The slot card also gains a "Default" row showing the value readably (`planned`, `8080`, `"svc"`, `true`).
- **`examples` now render as a card section.** Worked examples (`examples:`, a list of `value` / optional `description` objects) on a class, slot, enum, or type previously parsed and silently vanished. The entity card now shows an "Examples" section listing each value with its description when present, so a reader sees what a valid value looks like. LinkML `examples` has no standard RDF predicate, so it is documentation-only.
- **`aliases` and `see_also` editorial metadata now render and emit to RDF.** Alternative names (`aliases:`) and related-resource references (`see_also:`) on a class, slot, enum, or type previously parsed and silently vanished. The entity card now shows an "Aliases" row of comma-joined synonyms and a "See also" row of links — each `see_also` URIorCURIE is CURIE-expanded into a hyperlink, falling back to plain text when its prefix isn't declared. The generated RDF emits one `skos:altLabel` per alias and one `rdfs:seeAlso` per reference on the class or slot IRI, so synonyms and cross-references are discoverable both in the docs and downstream.
- **`deprecated` lifecycle metadata now renders and emits to RDF.** Marking a class, slot, enum, type, or the schema itself `deprecated: "<note>"` previously parsed and silently vanished. The entity card now shows a "Deprecated" badge with the migration note (on slots the badge rides the Characteristics row), and the generated RDF emits `owl:deprecated true` on the class or slot IRI — so a sunset element reads as sunset in the docs and downstream consumers can detect it.
- **Numeric value bounds on slots now show on the card.** A slot with `minimum_value` / `maximum_value` (e.g. a `strength` bounded `0.0..1.0`) previously parsed and vanished; the bounds now render as `≥`/`≤` badges on the slot card, read distinctly from the `min..max` cardinality badge. (Emitting them as an `owl:withRestrictions` datatype facet in the RDF — for reasoner-checkability — is a tracked follow-up.)
- **OWL relationship characteristics on slots now render and emit axioms.** A slot declared `symmetric`, `asymmetric`, `reflexive`, `irreflexive`, or `transitive` (LinkML's relationship metaslots) previously parsed and silently vanished. Each set characteristic now shows as a badge on the slot card *and* emits the matching `owl:<Name>Property` `rdf:type` axiom in the generated RDF — so a reasoner can use, e.g., `owl:TransitiveProperty` to infer the transitive closure. The object-vs-datatype property type is emitted as before; characteristics sit alongside it.
- **HTML docs gain Enumerations and Types sections.** The rendered doc previously had only Classes / Slots / Individuals, so enums and types — which the schema graph already draws as diamond and rectangle nodes — were invisible in the doc body. Now an **Enumerations** section renders one card per enum with its description and each permissible value's text, description, and `meaning` (a concept IRI hyperlinked with its cached upstream label), and a **Types** section renders one card per type with its parent type, `uri`, and `pattern`. Both gain sidebar nav entries with counts, `[[Name]]` cross-references resolve to the new `#enum-<name>` / `#type-<name>` cards, and the schema graph's hover reuses these rendered cards for enum and type nodes — so every node kind the graph draws now has a matching doc card, and the hover can't drift from the page. Sections are omitted when the schema declares no enums or types.
- **Schema graph hover cards surface more of what you authored.** Hovering a slot now shows its validation `pattern` (truncated, full regex on tooltip), an `identifier` flag, and the member ranges of a polymorphic `any_of`. Hovering an enum shows each value's description on tooltip (with its `meaning` IRI carried for future linking). IRIs in hover cards are now always fully expanded from their prefix; a `?` marks any whose prefix the schema didn't declare.
- **Graph notation legend.** A new `Legend` control in the graph toolbar toggles a key that explains the notation: node shapes (circle = class, pill = slot, diamond = enum, rectangle = type, dashed = abstract), edge styles (hollow triangle = is_a, dashed = mixin, plus domain / range / inverse / type-of), and the four crow's-foot cardinality terminators. The key is drawn by the same rendering code as the graph itself, so it always matches what's on screen. It remembers whether you left it open or closed, and starts collapsed on narrow screens.
- **`is_a`-heavy schemas auto-default to the Hierarchical layout.** When the manifest doesn't pin `html_default_layout` and you haven't picked a layout, the graph now chooses its default from the schema's inheritance density: if at least half the edges are `subclass_of` / `mixin` (a taxonomy-shaped schema), it opens in Hierarchical, which shows the tree legibly; otherwise it stays on SGD. A pinned `html_default_layout` or a remembered picker choice still wins.
- **Graph nodes are shaped by kind.** Node kind is now encoded by shape as well as color (so the graph reads in grayscale, per the visualization ADR): classes are circles, datatypes rectangles, enums diamonds, and slots pills. Abstract classes get a dashed outline instead of the old faint-fill cue, matching their "abstract" card badge. Shapes stay within the node radius, so hover/click/drag are unchanged.
- **Schema graph is more legible for multi-component schemas.** You can zoom in 40× (was 10×), and edge glyphs (arrowheads, crow's-feet) scale with zoom. Node and edge labels are sized in proportion to their node, so they grow as you zoom into a cluster and shrink — then drop out — as you zoom out, instead of pinning to a fixed screen size. Disconnected nodes (separate `is_a` trees, and isolated classes with no relationships yet) pack into a compact grid rather than smearing into a horizontal line; the layout then stretches to fill the viewport's width instead of sitting in a narrow band, and the fit reserves room for the overlaid control bars so nodes never render underneath them. Parallel edges between the same pair of nodes — such as a slot's `domain` and `range` both pointing at one class — render as curves that share endpoints and bow apart, so their terminators (arrow vs crow's-foot) read as distinct relations instead of overlapping.
- **`range` edges show slot cardinality as ER crow's-foot notation.** A `range` edge's target end carries a crow's-foot terminator read from the slot's effective cardinality: `1..1` mandatory-one, `0..1` optional-one, `1..*` mandatory-many, `0..*` optional-many; bounds outside that vocabulary (e.g. `2..5`) render as a small `min..max` label. The foot replaces the filled arrow on range edges and is always shown (it's data, not a direction decoration). It reflects the slot's global cardinality; per-class `slot_usage` refinements stay in the hover/class card.
- **External-term tooltips show every upstream annotation, not just one.** A reused term's definition, description, comment, and example are all surfaced in the hover tooltip (each on its own line), instead of a single priority-picked value. This fixes cases like CiTO, which puts the real definition in `rdfs:comment` and an `"Example: …"` in `dc:description` — the tooltip previously showed only the example. The link target remains the canonical term IRI.
- **Schema-graph edges are now drawn per kind, directed.** Following [a new ADR on graph-visualization conventions](docs/adr/005-graph-visualization-conventions.md), each edge kind renders distinctly instead of as an identical gray line: `is_a` is a solid line with a hollow triangle pointing at the parent (UML generalization), `mixin` is dashed with a hollow triangle, `domain`/`range`/`type_of` are solid with a filled arrow, and `inverse` is dashed with a filled arrow at both ends. Each kind also has its own muted color, but the distinction is redundant — line style and arrowhead shape carry it in grayscale too. Arrowheads sit on the target node's perimeter and scale with the node, and an "Arrows" toggle in the graph controls hides the heads for dense graphs (default on, persisted across reloads).
- **Graph hover card shows each slot's effective shape.** Hovering a class now lists its slots one per line with the resolved range and cardinality — `wasGeneratedBy: QuestionFormation (required)` — using the post-`slot_usage` view, so a slot refined on a subclass no longer shows its global un-refined range. Inherited entries carry their origin inline, explicit `minimum_cardinality` / `maximum_cardinality` bounds render as `min..max`, and slot-node hovers replace the old "Flags" row with a "Cardinality" row that includes the bounds. A consistency test pins that the class card and the hover payload in the same generated page agree on the effective range of a refined slot.
- **Every writer now says where an inherited slot came from.** The shared resolver tracks per-slot provenance — defined here, inherited via an `is_a` chain, flattened in from a mixin, or refined by `slot_usage` — and all three writers surface it: the HTML class card tags inherited slots with a muted "from `<origin>`" chip (e.g. "from mixin Named", "from Identifiable via mixin Auditable") and renders them compactly — the slot's description shows inline only on the defining class's card, and as a hover tooltip on inheriting cards, so a parent's prose isn't repeated across every subclass. Generated Rust structs carry an "Inherited from …" doc line on flattened fields, and the graph hover card's Slots row suffixes inherited entries with "(from …)". Slots a class declares itself stay untagged and keep their inline descriptions. When a slot is reachable both through `is_a` and a mixin, the `is_a` path is reported, deterministically.
- **Slot cardinality in the HTML class card honors explicit bounds.** A slot declaring `minimum_cardinality` / `maximum_cardinality` now renders its required/multivalued framing from those bounds (`min >= 1` → required, `max > 1` → multi-valued) instead of only the bool flags, with each flag still the fallback when its bound is absent. The reconciliation lives in one shared resolver view, so every writer that displays cardinality gives the same answer.
- **Classes section renders the `is_a` hierarchy as a tree.** Class cards now nest under their `is_a` parents in semantically nested lists, so the inheritance structure is visible in the doc body itself — no more reading each card's "Subclass of" line or switching to the graph view. Classes without a parent root the tree; classes with neither parent nor descendants sit flat alongside it; mixins don't create tree edges (they remain surfaced via each card's "Mixes in" row), and a class under both an `is_a` parent and mixins appears exactly once, under the parent. Each tree level is itself a card grid — leaf siblings tile across the available width while a class with its own subtree spans the full row — and child groups hang off a class-colored spine so the superclass → subclass connection reads at a glance. A Tree/Flat toggle above the cards switches back to the alphabetical grid, with the preference persisted across reloads. Anchor links keep working in both views — each class still renders exactly one card.
- **Upstream ontology labels and definitions in rendered docs.** External references — `subclass_of` grounding, cross-ontology mappings — now render with the upstream term's human-readable name instead of an opaque CURIE: `Information Content Entity` where the source says `cco:ont00000958`. Hovering shows a tooltip with the CURIE = IRI identity line plus the upstream definition when one is published (`skos:definition`, `IAO:0000115`, `dc:description`, or `rdfs:comment`, in that priority). Labels come from the upstream ontologies themselves: on first generate, panschema fetches the RDF for each declared prefix it recognizes (BFO, RO, IAO via OBO PURLs; CCO; PROV-O; SKOS; Dublin Core terms; CiTO; OA), extracts `rdfs:label` / `skos:prefLabel` per term, and caches the result on disk — repeat generates are fully offline. Everything fails open: a fetch error, an unknown prefix, or a missing label just renders the CURIE as before, and `--offline` skips fetching entirely (cached labels still render). Both `panschema generate` and `panschema publish` carry the labels. A `[label_sources]` section in `panschema.toml` / `panschema-publish.toml` maps a prefix name to a custom source URL — overriding the built-in source for that prefix (e.g. pinning a versioned release document for reproducible labels) or enabling labels for a prefix outside the built-in ecosystem entirely. `panschema generate --refresh-labels` evicts the cached label maps for the schema's sources and refetches them, for picking up a new upstream release.
- **Abstract classes now carry a small `abstract` badge in the HTML class card heading.** Authors marking foundational classes `abstract: true` (e.g. BFO/CCO bases that exist for inheritance but aren't meant to be instantiated) get a visible indicator at a glance — uppercase, muted color, inline next to the class title. Concrete classes render unchanged. Previously `is_abstract` reached only the graph-viz JSON; the persistent HTML view treated abstract and concrete classes identically.
- **Schema graph hover card surfaces resolved-schema context.** The hover card now answers the questions an author actually asks while reading the graph: for a class, which slots can it have, which parents (`is_a`) does it inherit from, and which mixins flatten into it; for a slot, which class(es) it's a domain of, its range, and whether it's required and/or multi-valued; for an enum, what permissible values it accepts. Lists show every entry; the full content is also available on click-to-pin in the persistent details panel. Edge hovers gain a one-sentence semantic blurb keyed off the edge kind (subclass-of, mixin, domain, range, inverse-of, type-of) for authors still building intuition for LinkML's relationship vocabulary. Resolution happens in the writer (walking `is_a` chains, mixins, `attributes:`, and `slots:` once per class) so the visualization layer never has to traverse the schema itself.
- **Schema graph hover-driven focus mode.** Hovering a node in the rendered schema graph brings the focal node, its 1-hop neighbors, and its 2-hop neighbors to full opacity; everything else dims to ~25%. The local subgraph jumps out instantly — the "what's near X" answer without needing to click or zoom. Restored on hover-out via the `mouseleave` handler — no fade animation, visual snap is the right read. A "Focus on hover" toggle button sits in the graph controls strip alongside Labels / Nodes / Edges; the preference persists across reloads. The neighborhood is computed once at hover-target-change time via a BFS frontier expansion (O(visited × avg_degree), sub-millisecond for any schema in practice) and cached on the interaction state, so per-frame rendering is an O(1) set clone instead of an O(E × hop_depth) re-walk. Touch users still get the existing tap-to-select flow unchanged. Composes with the hover details card — both fire together on a single hover.
- **Schema graph hover-driven details card.** Hovering a node or edge in the rendered schema graph surfaces an ephemeral mini-card anchored at the cursor with the entity's label, type (with an "(abstract)" suffix for abstract classes), schema-internal ID, IRI when one is declared, connection count, and the LinkML `description:` as a wrapped paragraph at the bottom. The card hides on hover-out, when no hover target is under the cursor, when the same node is already click-selected (the persistent details panel below already shows the same content), and on touch-coarse input — desktop-only by design. Positioning is viewport-relative with auto-flip near the right and bottom edges, so the card stays visible regardless of where the cursor is. Edge hovers render a `<source-label> ↓ <edge-type> ↓ <target-label>` triple. The existing click-to-select flow is preserved unchanged — click still pins the persistent panel.
- **Schema package manager.** Cargo-style dependency management for LinkML schemas. Every schema dependency is a "package": a directory containing `panschema-publish.toml` plus the main schema file it references at `[files].main`.
  - **Publishing standard**: `panschema-publish.toml` lives at the package root and declares the schema's authoritative name, version, LinkML target version, and main-file location. Schema authors publish through this file; consumers verify against it.
  - **Manifest**: `panschema.toml` in the consumer project declares `[schemas.<name>]` dependencies and per-schema `[generate.<name>]` codegen config. Cargo-style discovery (walk up from CWD).
  - **Lockfile**: `panschema.lock` records resolved version + SHA-256 checksum of each schema's main file. Committed alongside `panschema.toml`. Drift detection covers both checksum and version. (A `revision` field is reserved for future provenance; populated only when a source exposes a stable commit identifier — currently always `None`.)
  - **Source protocols**: `path:` for local packages, `github:owner/repo` for tagged GitHub commits. Both go through the same package model. Other protocols (`gitlab:`, `zenodo:`, `https:`) deferred to later releases.
  - **Github source**: anonymous tarball fetch from `codeload.github.com/<owner>/<repo>/tar.gz/refs/tags/v<version>` (no API rate limit). The tarball's top-level directory is `<repo>-<version>/`; the cache extracts to `~/.cache/panschema/github/<owner>/<repo>/<version>/<repo>-<version>/` with `fs2` file locking. Pluggable `TarballSource` trait for future protocols and for tests. Skips `pax_global_header` pseudo-entries that GitHub codeload includes. Symlink hygiene refuses paths that escape the extracted directory.
  - **Commands**:
    - `panschema init` — producer-side scaffolding. Writes a `panschema-publish.toml` in CWD. Three input modes: explicit flags (`--name X --version Y --main schema.yaml`), `--from <linkml.yaml>` (pre-fills name + version from the LinkML file's metadata), or no args (defaults to CWD basename + `0.1.0` + `schema.yaml`). Refuses to overwrite an existing publish file; `--force` opts in. After writing, prints a per-field provenance summary (explicit / from `--from` / default) so the user can see where each value came from. Post-write validation warns if the main file is missing or doesn't parse but still writes the publish file.
    - `panschema release` — producer-side version bump, modeled on `cargo release`. `--level patch|minor|major` does literal semver bumps (`0.x.y --level major` → `1.0.0`); `--version <x.y.z>` sets an exact value. Default behaviour is bump-only — prints the suggested git commands so the user can complete the release manually. `--git` runs `git add` + `git commit -m 'release: v<ver>'` + `git tag -a -m 'release v<ver>' v<ver>` (annotated tags, the only kind `git push --follow-tags` pushes); refuses on a dirty working tree or an existing tag. `--push` (requires `--git`) also runs `git push --follow-tags`. `--dry-run` prints the plan without writing or running anything. Refuses no-op bumps (`--version <current-version>`) with a clear "tag manually" hint. Refuses to release while the LinkML main file's `version:` field disagrees with publish.toml's `[schema].version`. Manifest edits go through `toml_edit` so comments survive.
    - `panschema add <spec>` — single positional spec, either `github:owner/repo@version` or a filesystem path to a package directory. Schema name is inferred from `panschema-publish.toml`; `--name <alias>` overrides. Writes only the `[schemas.<name>]` entry — the `[generate.<name>]` block is the user's to add when they want codegen output. (`generate` prints a clear "no `[generate.<name>]` block; skipping" hint for any schema without one.) Always re-runs `fetch` afterward so cache + lockfile stay consistent. Idempotent on same-shape adds; conflicting version or source raises a clear error rather than overwriting.
    - `panschema fetch` resolves all manifested schemas, populates the cache, and writes `panschema.lock`. Re-fetch is a no-op when the cached version is already extracted.
    - `panschema verify` re-checksums against the lockfile and errors with a clear diff on drift (catches both "schema edited but generate not re-run" and "publish.toml version bumped").
    - `panschema generate` (no `--input`) discovers the manifest and fans out across every populated writer key in each `[generate.<name>]` block (currently `html` and `rust`). `--input <file>` continues to work as a no-manifest shorthand for raw schema files.
  - **CLI ergonomics**: `SchemaSpec` parser (clap `FromStr`) catches malformed input at parse time — invalid version, unknown protocol, empty spec — before any side effects. Manifest edits go through `toml_edit` so user comments, key order, and whitespace survive.
- **`panschema publish` — multi-version HTML doc orchestration.** New subcommand that reads a `[publishing]` section from `panschema-publish.toml` and builds per-version HTML docs side-by-side, with an in-page version dropdown and banners that announce drift between the viewed version and the cohort's `current`.
  - **Manifest extension**: `[publishing] versions = [...], edge = "...", current = "...", url_pattern = "...", output_dir = "..."`. Parse-time validation rejects a `current` that isn't in `versions` and doesn't equal `edge`, with an error that names the legal alternatives.

Edge ref builds to `<output_dir>/<edge-name>/`. `<output_dir>/current/` is a byte-equal copy of the configured version's output, not a symlink (static hosts handle directories cleanly).
  - **Resolves all refs up-front**: a bad tag fails fast with a single combined error listing every unresolvable ref, before any partial build state lands on disk.
  - **In-page UX**: header gains a `<select>` populated from the cohort, default-selected to this page's version, with an inline `onchange` handler that substitutes `{version}` in `url_pattern` to navigate. Edge entries are badged `(edge)`. A non-current page shows a stale banner with a link back to `current/`; the edge page shows a distinct "edge build from HEAD" banner.

Tagged versions are unaffected by the flag.

Explicitly setting `url_pattern` in the manifest still uses the value verbatim for consumers who need an absolute form.
- **Schema graph layout picker.** The HTML viz now exposes a `<select>` next to the 2D/3D toggle for choosing which layout algorithm produces node positions.
  - **Force-directed (default)** — the in-tree CPU simulation tuned for viewport filling.
  - **Kamada-Kawai** (2D only) — energy-minimization layout via [egraph-rs](https://github.com/likr/egraph-rs)'s `petgraph-layout-kamada-kawai`. Produces visibly nicer node spacing on medium graphs (≤500 nodes) than force-directed at the cost of higher init latency; labelled "(slower init)" in the picker.
  - **SGD** (2D only) — egraph-rs's `petgraph-layout-sgd` (`FullSgd` variant). The modern stochastic-gradient variant of stress majorization: converges in `O(N · iters)` instead of `O(N² · iters)` and typically produces visibly comparable output. The recommended default for medium-to-large graphs. The internal RNG is seeded with a constant so the output is byte-identical across runs — load-bearing for `panschema publish`'s idempotent-republish guarantee. Same per-component shelf-packing as Stress for disconnected inputs. Labelled "(fast quality)" in the picker.
  - **Stress majorization** (2D only) — egraph-rs's `petgraph-layout-stress-majorization`. The algorithm behind graphviz's `neato -Kstress`; converges on a layout that minimizes the squared deviation between Euclidean and graph-theoretic distances. Produces cleaner cluster separation and more uniform edge lengths than force-directed on graphs with natural clusters (e.g. schemas with BFO/CCO/OA grouped classes). Disconnected components are laid out independently and shelf-packed into a roughly-rectangular bbox that targets the configured aspect, with per-component gap scaled to the largest component's dimension. Labelled "(cluster separation)" in the picker.
  - **Hierarchical (Sugiyama)** (2D only) — layered layout via [rust-sugiyama](https://github.com/paddison/rust-sugiyama) over the `is_a` / `mixin` sub-DAG. Property edges (range / domain / inverse / typeof) deliberately don't participate in layering — they overlay the layered output afterward, so cyclic property graphs (e.g. `Person.owns: Asset`, `Asset.owner: Person`) don't break the layered render. Pathological cycles in the hierarchy spine itself fall through to rust-sugiyama's internal greedy feedback arc set. Orphan nodes (no hierarchy edges) fall back to a grid below the layered region so the connected cluster keeps the central viewport. Labelled "(best for class hierarchies)" in the picker.
  - **Mode-aware**: 3D mode greys out non-force-directed options with a "(not implemented)" suffix since the WebGPU path only runs force-directed. A 2D-only preference (KK / Hierarchical) round-trips through localStorage without being overwritten on the 3D toggle, so toggling 3D → 2D restores the original choice.
  - **Manifest plumbing**: `panschema.toml` accepts `html_default_layout = "<algorithm>"` under each `[generate.<name>]` block, validated at manifest parse time against the canonical identifier list. Unrecognized layouts produce an actionable error rather than silently falling back.

- **Rust types writer.** `[generate.<name>]` now accepts a `rust = "<path>"` key alongside `html`, and `panschema generate` fans out across every populated writer per schema. The writer emits a single flat Rust module per schema with: structs for concrete (leaf) classes; marker traits for classes used as `is_a` parents or mixins, with supertrait bounds following the LinkML inheritance chain and `impl Trait for Struct {}` blocks per concrete descendant; flattened slot inheritance (parent + mixin slots merge into the consuming struct) with `slot_usage` overrides applied per-class; LinkML enums as Rust enums with `#[serde(rename)]` on variants whose text isn't a valid Rust identifier; primitives mapped to `String` / `i64` / `bool` / `f64` / `chrono::DateTime<Utc>`; `Option<T>` / `Vec<T>` framing for optional / multivalued slots; polymorphic `any_of` ranges as per-slot `#[serde(untagged)]` union enums (branches without an explicit `range:` inherit the slot's outer range); trait-only-class slot ranges as `<Name>Kind` closed enums of concrete descendants, with `String` fallback + breadcrumb comments when a trait class has no concrete descendants; `Box` wrapping for class-typed single-valued fields to break recursive layout cycles; doc-comments from LinkML descriptions; per-field `#[serde(rename = "...")]` so the LinkML wire format round-trips. Ergonomic derives: every struct gets `PartialEq`, plus `Default` when all fields are conservatively default-able (`Option<T>` / `Vec<T>` always count; required primitives that implement `Default` qualify too). Every emitted enum carries `#[non_exhaustive]` so adding permissible values or subclasses to a schema doesn't break downstream `match` statements. Robust against malformed inputs: circular `is_a` / `mixin` chains terminate cleanly via visited-set guards rather than overflowing; unresolved global slot references emit `// WARNING:` comments rather than silently dropping fields. Generated code depends on `serde` and `chrono` in the consumer's `Cargo.toml`. Renderers are generic over `std::fmt::Write`, so `RustWriter::render_into(&mut sink, schema)` streams directly into any sink without an intermediate `String` allocation; `render(schema) -> String` remains as the convenience wrapper. Recursive `Eq + Hash` analysis: a struct, `<Name>Kind` closed enum, or `any_of` union derives `Eq + Hash` only when every transitive field type does (`f64` family disqualifies; `chrono::DateTime<Utc>`, `NaiveDate`, `NaiveTime` qualify; LinkML enums always qualify). Self-recursive class fields via `Box<T>` preserve the inner trait set; the fixpoint terminates because the support bit only flips from `true` to `false`. Each concrete struct also gets a `pub fn new(<required_fields…>) -> Self` constructor (optional fields default to `None`, multivalued to `Vec::new()`) so consumers survive schema additions of optional fields without breaking calling code; structs with no required-single fields skip the constructor since `Default::default()` already covers the empty-arg case.
- **Generated Rust is excluded from the consumer's formatter and linter rather than formatted.** Each generated module now opens with `#![cfg_attr(rustfmt, rustfmt_skip)]` and a `clippy::all` allow (alongside the existing `// @generated` marker), so `cargo fmt` and `cargo clippy` leave it untouched with no `rustfmt.toml` or clippy configuration on the consumer's side. The generated bytes are committed verbatim, which is what lets a drift check (re-generate and byte-compare) match a committed file: panschema does not format its output, and the formatter does not rewrite it, so the two can never disagree over whitespace. (The raw layout is intentionally not rustfmt-canonical; it is meant to be read as generated, not hand-tuned.)

- **MSRV bumped from 1.85 to 1.88** to enable let-chain syntax (`if let X = y && cond`) in source

### Fixed
- **OWL/Turtle output now declares the standard `xsd:`/`rdf:`/`rdfs:`/`owl:` prefixes it uses.** A datatype range like `integer` emitted `rdfs:range` with a verbose absolute `http://www.w3.org/2001/XMLSchema#integer` IRI because the OWL writer's prefix map carried only the schema's own `prefixes:` block, never the built-ins the graph itself emits. Both Turtle-emitting writers (OWL and SHACL) now build their prefix map through one shared helper that layers each writer's built-in prefixes on the schema's, so `xsd:`-typed terms serialize in compact form.
- **`generate --output new/dir/out.<ext>` now creates missing parent directories for every format.** Only the Rust and Postgres writers created the output's parent directory; the OWL/Turtle, JSON-LD, RDF/XML, N-Triples, graph-JSON, and SHACL writers called `File::create`/`fs::write` straight away and failed with "No such file or directory" when the target directory didn't exist. All file-producing writers now go through one shared `ensure_output_parent` helper, so the behavior is uniform across formats.
- **SHACL rule shapes are now correct for integer equality, unknown slots, and one-sided rules.** Three fixes to the conditional `rules` → SHACL projection: (1) an `equals_number` check on an integer-range slot emitted a `sh:hasValue "N"^^xsd:double` literal, but `sh:hasValue` is datatype-sensitive term equality and the instance data is `xsd:integer`, so the literal could never match — silently inverting the rule; the literal is now typed to the slot's range (`xsd:integer` for integer ranges). (2) A rule condition naming a slot the class doesn't have previously fabricated a `{ontology}#{slot}` property IRI and emitted a shape that rejected all valid data; such a rule (and a one-sided or empty-conditioned rule) is now skipped, with a `generate --format shacl` warning naming it — matching how the Postgres writer already reports unrepresentable rules. (3) A shared skip-reason check keeps the writer's emission and the warning in agreement.
- **Generated Rust ends with a single terminating newline.** The writer previously emitted a trailing *blank* line (`}\n\n`), which EOF-normalizing pre-commit hooks rewrite and which would defeat a regenerate-and-`git diff` drift check (a fresh generation never matched the normalized committed file). Output now ends in exactly one `\n`, so it commits verbatim and diffs cleanly.
- **OWL/Turtle reader now reads `owl:deprecated`, `skos:altLabel`, `rdfs:seeAlso`, the SKOS mapping predicates, and OWL property characteristics back into the IR.** The writer already emitted these for deprecated, aliased, cross-referenced, mapped, and characteristic-bearing classes and slots, but the reader never parsed them — so a Turtle → IR → Turtle round-trip silently dropped every one of them. The reader now reads `owl:deprecated true` into the `deprecated` flag, `skos:altLabel` into `aliases`, `rdfs:seeAlso` into `see_also`, `skos:exactMatch`/`closeMatch`/`relatedMatch`/`narrowMatch`/`broadMatch` into the matching `*_mappings`, and `owl:SymmetricProperty`/`AsymmetricProperty`/`ReflexiveProperty`/`IrreflexiveProperty`/`TransitiveProperty` into the corresponding characteristic flags — so these constructs survive a Turtle round-trip instead of vanishing on read-back. (RDF carries only the deprecation boolean, so the migration-note text isn't recovered, but the deprecated state is.)
- **Generated Rust code now escapes reserved keywords — including type names.** A schema with a name landing on a Rust keyword — an enum permissible value like `virtual`, a slot named `type` or `move`, or a class or enum *named* `type` or `move` — previously emitted the bare keyword, producing code that didn't compile (`error: expected identifier, found reserved keyword`). This now covers every emission site: enum variants, struct fields, constructor parameters, and the struct / enum / trait type names themselves (a class named `move` becomes `pub struct r#move`, referenced as `r#move` everywhere it appears — field types, `Box<…>` inners, trait bounds, and `impl` targets). Keyword idents are emitted as raw identifiers (`r#type`, `r#virtual`); the handful that are illegal even as raw identifiers (`crate`, `self`, `Self`, `super`, `_`) get a trailing-underscore mangle instead (`self` → `self_`). An escaped field or variant keeps its original wire name via a `#[serde(rename = "…")]` so JSON round-trips unchanged; type names aren't serialized, so they need no rename. Separately from keywords, a name that cannot become a Rust identifier at all (spaces, punctuation, a leading digit) now fails the write with an error naming the offender — previously it was emitted verbatim, producing code that didn't compile or, for a hostile schema, injected arbitrary tokens into the generated source.
- **Per-class `slot_usage` range narrowing now shows on the class card and in the graph.** A class that narrows an inherited slot via `slot_usage` — a single `range`, a smaller `any_of`, or `maximum_cardinality: 0` — previously showed the wide *inherited* union everywhere (the scalar-range-over-`any_of` case left both populated, so the union masked the narrowing), hiding the per-class shape. The class card now renders the induced range — a scalar narrows to that one range, a smaller union replaces the inherited one, and a slot suppressed by `maximum_cardinality: 0` reads as "has no value" — and the schema graph draws range edges from the narrowing class directly to its induced targets (and none for a suppressed slot), so a class connects to exactly the artifacts it actually takes and produces instead of the whole envelope. Computed once in the shared resolver (`InducedRange`) so every writer consumes the same per-class view.
- **Polymorphic `any_of` ranges are no longer dropped.** A slot whose range is an `any_of` union (e.g. an input that may be a `Dataset`, `Question`, or `Result`) now draws a graph range edge to every member and lists the whole union on its slot card. Previously the range vanished — the member classes appeared disconnected in the graph and the card's Range row was blank.
- **Graph node detail cards label slot nodes correctly.** A slot node's hover/detail card now reads `Type: Slot` (and `Enum` / `Type` for those kinds) instead of `Class`. The label is read from the node's kind, not inferred from a color channel that collided between the class and slot palettes.
- **External `subclass_of` grounding now round-trips through the IR, HTML, and RDF.** LinkML's `subclass_of:` field — used to declare `rdfs:subClassOf` to upstream ontology classes (BFO, CCO, IAO) distinct from intra-schema `is_a` — previously parsed and then vanished: no IR field existed, the HTML showed nothing, and the emitted RDF was missing the grounding axioms the schema declared. The class card now renders a "Subclass of (external)" row with CURIE-expanded hyperlinks (and a muted-text fallback for unrecognised prefixes), and the RDF serializers emit one `rdfs:subClassOf <expanded>` triple per entry.
- **`class_uri` and `slot_uri` in the HTML class and slot cards are now CURIE-expanded hyperlinks.** Previously these rendered as plain `<code>` text, so a `class_uri: cco:ont00000005` was visible but not clickable, and a CURIE-only entry like `cco:ont00000005` displayed the abbreviated form with no path to the upstream definition. The cards now wrap the displayed text in `<a href="…">` pointing at the expanded IRI (resolved against the schema's `prefixes:` table), and the existing copy-IRI button copies the expanded form so a paste yields a directly-resolvable URL. CURIEs whose prefix isn't declared fall back to plain text, unchanged.
- **Cross-ontology mappings now round-trip through the IR, HTML, and RDF.** `exact_mappings`, `close_mappings`, `related_mappings`, `narrow_mappings`, and `broad_mappings` on classes and slots previously parsed and then vanished — the YAML accepted the keys but no field existed on the IR, so the grounding values authors wrote (BFO / CCO / CiTO / IAO IRIs) silently dropped from every output. The five fields are now first-class IR fields; HTML class and slot cards render a "Mappings" row with the mapping kind and a CURIE-expanded hyperlink (falling back to a muted-text indicator when the prefix isn't declared); the RDF serializers emit one triple per mapping under the matching SKOS predicate (`skos:exactMatch`, `closeMatch`, `relatedMatch`, `narrowMatch`, `broadMatch`).
- **Schema graph hover card now surfaces `slot_usage`-introduced slots.** Class-node hover previously walked `is_a` + mixins + `attributes:` + `slots:` but missed slots that a subclass added or refined via `slot_usage`. Now goes through the shared resolver, so e.g. `Question` (extending `Activity`) shows the `wasGeneratedBy` slot it refines via `slot_usage` even though the slot was inherited.
- **Static graph layouts (Kamada-Kawai, Hierarchical, Stress majorization) now survive node selection and single-node drag.** The drag handler in the visualization reheated the simulation unconditionally on every mousedown that hit a node — including click-without-drag — which re-enabled the per-tick force-directed physics. The result: any click on a KK / Sugiyama / Stress page sent the layout into force-directed motion, undoing the static algorithm's work. Static layouts now skip the reheat on drag, so a single-node drag moves only that node and the rest of the layout stays where the static algorithm placed it.
- **Graph viz no longer drops slots referenced only via `class.slots:`.** The graph builder previously emitted class↔slot edges only from the slot-side `domain:` field, so slots listed by a class but missing `domain:` rendered as orphan nodes — even though the HTML class card already drew the connection correctly. The builder now also walks each class's `slots:` list and emits the equivalent edge, deduping against the slot-side pass so a slot with both `slot.domain = C` and `C.slots: [s]` produces one edge, not two. LinkML treats `domain` and `domain_of` as the same relation; the graph now matches.
- **Schema-level description in the metadata card is no longer double-escaped.** The metadata card template mounted `comment` without `|safe`, but the writer pre-renders that value as HTML (through `render_description`) — so Askama escaped the writer's `<p>…<a href="…">…</a>…` markup a second time and authors saw literal `<p>…</p>` markup as visible text. The fix is `{{ comment|safe }}`, matching the entity-card descriptions.
- **Header brand link in rendered HTML is no longer an absolute `/`.** Each page's `<a class="site-title">` previously emitted `href="/"`, which resolved to the *domain root* on subpath deploys (e.g. clicking the brand on `https://<user>.github.io/<repo>/schema/main/` sent the user to `https://<user>.github.io/`, 404 for the project). `panschema generate` now emits `./` (the page lives at the output root, so `./` is the deploy root). `panschema publish` reads the target from a new manifest field `[publishing].site_root_url`, default `"../current/"` — parent-relative to the canonical current-version page within the publish output cohort, symmetric with `url_pattern`'s `"../{version}/"` default. The default works for any standalone publish deploy; consumers whose publish dir is nested under a parent site (e.g. `<book>/schema/<version>/`) override the field (e.g. `"../../"` to escape into the book).
- **Schema graph 2D layout now uses the full configured aspect-ratio viewport and produces legible labels at all graph sizes.** The CPU force simulation gains three composable changes: (1) anisotropic axial centering (`forceX` / `forceY` with `gravity_y / gravity_x = (w/h)³`) so isolated nodes equilibrate at a radius matching the configured aspect — landscape containers no longer leave half the horizontal space empty; (2) the largest connected component is placed at origin so anisotropic gravity doesn't have to fight an off-center initial layout; (3) `link_distance`, `charge`, and `collide_padding` scale with `√N` so the same defaults produce visibly-legible layouts from 6-node fixtures up to 100-node ontologies — the collide-padding scaling matters most, since collide is the only force that enforces minimum geometric spacing between every node pair. Validated via a multi-scale Playwright iteration test (`#[ignore]`-d for routine CI) that screenshots a synthetic graph at phone (390×844), laptop (1440×900), and 4K (3840×2160) viewports.
- **RDF serializers (TTL / JSON-LD / N-Triples / RDF/XML) now expand CURIE-shaped IRIs against the schema's `prefixes:` table** before emission. Previously a `class_uri: cco:ont00000005` produced `<cco:ont00000005>` — invalid as N-Triples (the spec requires absolute IRIs in `<...>`) and ambiguous in TTL/JSON-LD/RDF/XML (parsers read it as a relative IRI against the empty base). The TTL writer now also wires sophia's `TurtleConfig::with_prefix_map` so the schema's prefixes round-trip into `PREFIX` declarations at the top of the output. JSON-LD and RDF/XML emit fully-expanded absolute IRIs.
- **LinkML mixins now emit `rdfs:subClassOf` alongside the `is_a` parent in OWL output.** LinkML treats `mixins:` as multiple inheritance; the prior emitter only honored `is_a`, so a class declaring three mixins lost three subClassOf relations from its RDF representation.
- **RDF/OWL output now declares properties for inline `attributes:` and inherited/mixed-in slots.** The RDF family (TTL, JSON-LD, RDF/XML, N-Triples) walked only top-level `slots:`, so a class written in the idiomatic inline `attributes:` form — or one whose slots come via `is_a`/mixins — emitted as a bare `owl:Class` with none of its properties declared, and any SHACL `sh:path` pointing at those properties had no OWL counterpart. The writer now resolves each class's effective slots (the same resolution the HTML, Rust, Postgres, and SHACL writers already share), declaring every attribute/effective slot as a property with its type, label, and range plus an `rdfs:domain` naming the owning class — deduped so a slot that is both top-level and effective emits once.
- **RDF/OWL now expands a bare (colon-less) name against the schema's `default_prefix`, consistent with the HTML output.** CURIE expansion is now a single shared implementation used by the RDF builders and the HTML writer, so a `class_uri`/`slot_uri` written as a bare name resolves the same way in every format instead of the RDF writer leaving it unexpanded.
- **Range aliases (`int`/`bool`/`str`) map to the same type across every writer, and no writer fabricates an `xsd:` IRI for a non-primitive range.** A shared primitive table now backs the RDF/OWL, SHACL, Postgres, and Rust type mappings: a range written `int` emits `xsd:integer` (not a nonexistent `xsd:int`), and a range that names no class, enum, `types:` entry, or built-in primitive — a typo, already flagged by the dangling-reference diagnostic — emits no `rdfs:range`/`sh:datatype` at all instead of a fabricated `xsd:{name}`. This generalizes the earlier enum-range fix to every non-primitive range.
- **OWL/RDF output no longer fabricates a nonexistent `xsd:` datatype for an enum-ranged slot.** A slot whose range is an enum is neither an object property nor an XSD scalar, but the range emitter fell through to the scalar mapping and produced `rdfs:range xsd:{EnumName}` — a datatype that does not exist. Enums have no RDF projection yet, so such a slot now emits no `rdfs:range` at all (guarded the way the SHACL and Postgres writers already handle enum ranges) instead of a dangling reference to a made-up datatype.
- `YamlReader` now infers metaobject names from their dict keys (idiomatic LinkML), so explicit `name:` and permissible-value `text:` fields are optional. Applies to classes, slots, enums, types, class attributes, class slot_usage, and permissible values. Schemas produced by `linkml-runtime` and the broader LinkML toolchain (`gen-owl`, `gen-shacl`, `gen-python`) now load without modification. Explicit names still work; an explicit name that disagrees with the dict key is now a clear parse error.
- `GraphWriter` now emits range edges for inline class attributes (e.g., `Student.year` → `YearEnum`). Previously only top-level `slots:` produced domain/range edges, so most relationships in idiomatic LinkML schemas were silently dropped from the visualization. Inline attributes connect the owning class directly to the range target (no separate slot node), labeled with the attribute name.
- 3D camera `zoom()` direction was inverted relative to the 2D camera and the documented contract; `factor > 1.0` now zooms in for both.

### Security
- **Generated HTML docs no longer let schema content break out of the embedded graph JSON (stored XSS).** The interactive graph's data is embedded in an inline `<script>` as JSON, and JSON does not escape `<` — so a `</script>` inside any schema string (a class/slot name or `description`) closed the script element mid-JSON and executed whatever markup followed, in the browser of anyone viewing the generated (or published) docs. Every `<` in the embedded JSON is now escaped as `<`, which `JSON.parse` decodes back to `<` — the graph data is byte-for-byte unchanged, only the on-page representation is safe. Affected anyone rendering docs from a schema they did not author.
- **The dev server (`panschema serve`) now binds loopback (`127.0.0.1`) by default instead of all interfaces (`0.0.0.0`).** It previously exposed the generated docs to the entire local network while printing a `http://localhost:…` URL that hid the exposure. Pass `--host-all` to opt back into binding all interfaces.
- **`imports:` can no longer escape the schema's directory tree.** A relative import such as `../../../etc/passwd` was resolved and merged like any local file; a fetched, untrusted schema package could use it to pull arbitrary host files into a consumer's generated output. Every import (transitively) must now resolve within the root schema's own directory tree, canonicalized so `..` escapes are caught by where they actually point — an out-of-tree import is a hard error naming the offending entry.

- **Reader/Writer Architecture**: Extensible pipeline for multi-format support
